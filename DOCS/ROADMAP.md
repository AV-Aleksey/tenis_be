# Roadmap бэкенда (последовательность реализации)

## 1. Каркас приложения

- Инициализация NestJS, зависимости, Prisma + PostgreSQL, конфиг (`@nestjs/config`), ValidationPipe.
- Swagger/OpenAPI, health/readiness при необходимости.
- Глобальные фильтры исключений, единый формат ошибок.
- Первый эндпоинт smoke-test (`hello-world` / `GET admin/test` по правилам проекта).

## 2. Модуль профиля (без авторизации)

- CRUD профиля, `slug`, список по `id[]` (`POST /profile/list`) по `DOCS/tech/backend/profile.md`.
- Внутренние идентификаторы профиля без привязки к JWT; ограничение доступа к чужим профилям временно не делаем или заглушки.

## 3. Справочники и сиды

- Таблицы и сиды: роли, статусы профиля, статусы матча и др. по `DOCS/tech/database/ENTITIES.md`.
- Миграции Prisma, индексы по внешним ключам и частым фильтрам (`DOCS/tech/database/MAIN.md`).

## 4. Модуль challenge (матч / вызов)

- Сущность матча, статусы, поля формата 1x1 / флаг `is_team_challenge`, базовые переходы статусов.
- Сценарии из `DOCS/CHALLENGE_SYSTEM.md`: прямое приглашение; задел под open match (публикация без соперника / набор состава).

## 5. Связка профиль ↔ challenge — полный пользовательский флоу

- Создание вызова, ответ/истечение, закрытие создателем, внесение счёта, финализация.
- Антиспам: не создавать новый активный вызов той же паре при незакрытой игре (`CHALLENGE_SYSTEM.md`).
- Пересчёт рейтинга и `incrementTotalGames` / `updateRating` на профиле после финализации (`BUSINESS_GOAL.md`, шаги 10–11).

## 6. Расширение онбординга

- Опрос/калибровка стартового рейтинга (контекст Survey в `ENTITIES.md`), обновление `onboarded`, выдача стартового `rating`.
- Ветвление после базового профиля без блокировки ранних этапов разработки.

## 7. Парный формат (2 на 2)

- Полный флоу для `is_team_challenge = true`: составы, подтверждения участников, счёт по командам.
- Согласование полей `player_*`, `opponent_player_*` и правил завершения матча.

## 8. Вспомогательная функциональность к ядру

- Загрузка аватаров: Multer, локальное хранилище, ограничения типов/размера (`DOCS/tech/backend/stack.md`).
- Уведомления по ключевым событиям матча (сущность Notification, `ENTITIES.md`) — после стабильного матч-флоу.
- Спорные результаты / эскалация (если заложено в продукте) — после двухстороннего подтверждения счёта.

## 9. Авторизация и закрытие API

- **Два канала доступа** (можно вводить параллельно или по приоритету продукта):
  - **Классический REST** (телефон/Telegram как логин + пароль): модуль `auth-gateway`, JWT access/refresh, guards — как в `DOCS/tech/backend/auth_module.md`, `stack.md`.
  - **Telegram Mini App / WebApp**: отдельный контур — не «логин/пароль», а **подписанные данные клиента**. Клиент передаёт строку `Telegram.WebApp.initData`; бэкенд **обязательно** проверяет её на сервере по [алгоритму Telegram](https://core.telegram.org/bots/webapps#validating-data-received-via-the-web-app): HMAC-SHA256, секрет из токена бота и константы `WebAppData`, сравнение с полем `hash`; дополнительно проверка **`auth_date`** (например, отсечка по возрасту пакета), защита от replay. Допустимо после успешной проверки выдать **короткоживущую сессию** (например, свой JWT) чтобы не гонять криптографию на каждый запрос — но доверие к пользователю всегда опирается на валидный `initData` или на уже выданную сессию.
  - **Чистый бот (webhook/long polling)**: идентификация пользователя из **обновлений Telegram** (`from.id` и т.д.) при валидном webhook (секрет URL / проверка источника); отдельный JWT для этого сценария обычно не нужен — доверие к контексту даёт канал Telegram.
- Связка идентификатора доступа ↔ профиль (`telegram` / `auth_user_id` — по выбранной модели), единые guards/стратегии под канал.
- Удаление временных обходов «без auth» на профиле и матчах.

## 10. Админка и аудит (после стабильного приложения)

- Роль Admin, административные сценарии, журнал действий (`DOCS/ROLES.md`).
- Ручные правки результата, сезоны/глобальные настройки — по отдельным ТЗ при появлении требований.
